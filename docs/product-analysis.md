## 一、产品概述

**QGraph** 是一个**轻量级可视化 Pipeline 编排工具**，定位为在 Makefile（太简陋）和 Airflow（太重）之间，为**个人开发者和小团队**提供**轻量、可视化、零配置**的脚本编排解决方案。

### 核心价值主张

- **pip install 即开箱即用**，无需 Node.js、数据库等基础设施
- **可视化 DAG 编辑器**，拖拽节点、连线编排工作流
- **5 种节点类型**：Shell Command / Python Script / Python Function / Input / Output
- **并行执行**：基于 DAG 拓扑排序，无依赖节点自动并行
- **断点续传**：失败后可从失败节点处 Resume
- **AI 原生支持**：自动生成 `QGRAPH.md`，AI 编码工具可直接创建 Pipeline

### 目标用户

| 用户类型 | 典型场景 |
|----------|----------|
| ML 工程师 | 训练流水线（数据准备→训练→评估→导出） |
| 数据工程师 | ETL 脚本编排、定时任务 |
| 后端开发 | 部署脚本、迁移任务、批处理 |
| 科研工作者 | 实验流程自动化 |
| DevOps | CI/CD 流水线、自动化测试 |

### 🎯 差异化亮点

#### 1. Quick Add — 双击画布粘贴命令，自动解析创建节点

```
输入：python train.py --epochs 10
生成：Python Script 节点 (script_path=train.py, args=--epochs 10)

输入多行脚本：
  cd /data/project
  python prepare.py
  python train.py
  python evaluate.py

生成：4 个节点 + 自动串行连线
```

#### 2. Input 参数传递

```
[Input 节点]
  EPOCHS=10
  LR=0.001
  OUTPUT_DIR=./output

自动注入到所有下游节点的环境变量：
  - Python 脚本：os.getenv("EPOCHS")
  - Shell 命令：$EPOCHS (Linux) / %EPOCHS% (Windows)
```

#### 3. 项目级存储

```bash
qgraph init

# 创建 .qgraph/ 目录
my_project/
├── .qgraph/
│   ├── graphs/          # Pipeline 定义（可 git 管理）
│   ├── logs/            # 运行日志
│   └── running/         # 运行中状态
├── train.py
└── ...
```

### 📈 目标用户刚需程度

| 用户类型 | 场景 | 痛点 | 刚需程度 |
|----------|------|------|----------|

| **科研工作者** | 实验流程自动化 | 步骤多、易出错、难以复现 | ⭐⭐⭐⭐ |
| **DevOps** | CI/CD 流水线 | 需要可视化反馈和日志 | ⭐⭐⭐ |

### 🎯 刚需场景分析

#### ✅ 强刚需场景 1：ML 训练流水线

```
[Input] → [Prepare Data] → [Train Model] → [Evaluate]    → [Export ONNX] → [Output]
                                 ↓            ↓
                                 └── [Quantize] ──┘

痛点：
- 多个脚本需要按顺序执行
- 参数（epochs、lr）需要传递给多个脚本
- 训练完成后才能进行评估和量化（并行）
- 失败后希望从失败节点继续，不想重跑全部

QGraph 解法：
- 可视化 DAG 编排
- Input 节点参数自动注入下游
- 并行执行支持
- 断点续传（Resume）
```

#### ✅ 强刚需场景 2：多步骤部署脚本

```
[Build] → [Test] → [Package] → [Deploy] → [Verify]

痛点：
- Makefile 语法不直观
- 失败后难以定位问题
- 无法单独测试某个步骤
- 没有执行历史记录

QGraph 解法：
- 可视化编辑器
- 实时日志输出
- 节点自测功能
- 运行历史持久化
```

#### ✅ 强刚需场景 3：数据迁移任务

```
[Export Old DB] → [Transform] → [Import New DB] → [Validate]

痛点：
- 脚本依赖关系复杂
- 失败后难以恢复
- 需要手动记录执行进度
- 参数配置散落在多个脚本中

QGraph 解法：
- DAG 可视化依赖关系
- Resume 功能跳过成功节点
- Input 节点统一管理参数
- 日志自动持久化
```

#### ⚠️ 非刚需场景

| 场景 | 原因 | 替代方案 |
|------|------|----------|
| 简单定时任务 | cron 足够 | crontab / systemd timer |
| 超大规模分布式调度 | QGraph 不支持 | Airflow / Dagster |
| 强数据血缘需求 | QGraph 无资产概念 | Dagster |
| 多用户协作 | QGraph 无权限管理 | Airflow / Prefect Cloud |
